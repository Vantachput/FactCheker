AI & Search Services

The AI system core: interaction with LLM providers.

Architectural Decision: This module abstracts interaction with three different AI providers: 1. OpenAI (for search query generation and analysis via base models). 2. Together AI (for fast inference of fine-tuned Llama/Mistral models). 3. Perplexity API (for interactive real-time search and deep research).

Business Logic (Prompting & Pipeline): - The generate_search_query function converts an emotional user claim into

a factual search query (e.g. ‘Zelensky banned breathing’ -> ‘Zelensky order ban’).

  • Base GPT models (call_base_gpt) receive pre-filtered sources (verified vs unverified) in their context, forcing the AI to rely only on trusted data (RAG — Retrieval-Augmented Generation).

  • sonar family models (Perplexity) use their own built-in search engine. They are configured with strict system prompts to prevent English output and enforce journalistic fact-checking standards.

async services.ai_service.call_base_gpt(claim: str, verified_srcs: list[str], unverified_srcs: list[str], model_id: str, user_id: int) → str[source]

Analyzes a news claim using base OpenAI models with context (RAG).

Implements the logic: 1. Passes arrays of ‘Trusted’ and ‘Unverified’ sources. 2. Forces the AI to compare facts and find consensus across sources. 3. Determines the final verdict (TRUE, MANIPULATION, FAKE, UNCONFIRMED)

based on a Logic Matrix.

Parameters:

  • claim (str) – The user’s claim to verify.

  • verified_srcs (list[str]) – Strings from official and verified sites (A+, A, B whitelists).

  • unverified_srcs (list[str]) – Other general internet sources.

  • model_id (str) – Selected base OpenAI model (e.g. gpt-4o).

  • user_id (int) – User ID for usage statistics.

Returns:

Detailed analysis and verdict in Markdown format.

Return type:

str

async services.ai_service.generate_search_query(user_text: str, model_id: str = 'gpt-4o-mini') → str[source]

Generates an optimized search query from the user’s text.

The ‘Search Query Architect’ algorithm strips emotional coloring, adjectives, and specific (potentially fake) details, retaining only ‘Anchor Facts’ (dates, locations, proper names).

Parameters:

  • user_text (str) – Original message from the user.

  • model_id (str, optional) – Model to use for generation. Default is “gpt-4o-mini”.

Returns:

A short search query (4-7 words in Ukrainian). If an error occurs, returns the first 100 characters of the original text.

Return type:

str

Search service via Serper API with source filtering.

Architectural Decision: This module separates web-search logic from AI interaction. We use Serper.dev as a proxy for the Google Search API. The module’s key feature is the filter_sources mechanism, which implements the source reliability verification business logic.

Business Logic (Source Filtering Algorithm): 1. Results received from Google are verified by URL. 2. The parser extracts the domain name from the URL. 3. The domain is compared against ALLOWED_DOMAINS — a hardcoded “whitelist”

of resources (official government sites, global media at A+, A, B levels).

  1. Results are split into verified (whitelisted) and unverified (others), allowing the AI to base its verdict only on trusted sources.

services.search_service.filter_sources(results: list[dict]) → tuple[list[str], list[str]][source]

Splits search results into trusted and untrusted.

Each found result is compared against the ALLOWED_DOMAINS list. Results are formatted as text blocks for subsequent inclusion in the AI prompt.

Parameters:

results (list[dict]) – List of results from serper_search.

Returns:

Two lists of formatted results: (verified_sources, unverified_sources).

Return type:

tuple[list[str], list[str]]

Examples

>>> filter_sources([{'link':'https://bbc.com', 'title':'T', 'snippet':'S'}])
(['--- TITLE: T\nURL: https://bbc.com\nCONTENT: S\n'], [])
